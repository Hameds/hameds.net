<p>امروز هم توسعه برنامه فروشگاهی که به روش TDD نوشتیم را با نگاهی نزدیک‌تر به سرویس Order Fulfillment که یک سرویس خارج از برنامه اصلی است، ادامه می‌دهیم.</p><p><strong>ارسال سفارش</strong></p><p>اگر نوشته قبلی را خوانده باشید، به یاد دارید که ما با یک سرویس خارج از برنامه اصلی برای اجرای سفارشات کار می‌کردیم. آن‌ها یک API فراهم کرده‌اند و ما OrderFulfillmentService را فراخوانی می‌کنیم. interface این API چندین فراخوانی و مجموعه‌ای از قوانین برای ترتیب فراخوانی‌ها دارد. خلاصه‌ای از این interface را در کد زیر می‌توانید ببینید:</p><!--kg-card-begin: html--><script اجرایی ما از آن استفاده می‌کند و بنابراین ما هم باید از همین interface استفاده کنیم. چند قانون پیرامون گردش کار این API وجود دارد:</p><ul><li>کاربر باید برای هر سفارش از یک session استفاده کند</li><li>کاربر برای باز کردن session باید نام کاربری و کلمه عبور خود را ارائه دهد</li><li>همه فراخوانی‌های سرویس باید شامل شناسه (id) مربوط به session باشند</li><li>کاربر باید تایید کند که همه اقلام سفارش، قبل از ثبت سفارش در انبار وجود دارد. ثبت سفارش اقلامی که در انبار نیستند باعث شکست ثبت سفارش می‌شود</li><li>کاربر باید session را به جهت تکمیل سفارش خاتمه دهد</li><li>شناسه session کد رهگیری سفارش خواهد بود</li></ul><p>چند قید (constraint) در این قوانین وجود دارد که باید روی آن‌ها کار کنیم. در نوشته امروز به همه آن‌ها نخواهیم پرداخت، اما روی ترتیب عملیات مربوط به این قیدها کار خواهم کرد. اولین چیزی که نیاز داریم یک تست کیس (test case) است</p><p>وقتی که سفارشی ثبت می‌شود که در انبار موجود است، گردش کار اجرای سفارش باید کامل شود</p><p>درست است. این یک تست کیس بسیار ساده است. این تست روی هدف نوشته امروز بسیار متمرکز شده، که در واقع اطمینان از این مطلب است که فراخوانی‌های OrderFulfillment به ترتیب انجام شوند.</p><!--kg-card-begin: html--><script PlaceOrder از OrderService می‌سازم.</p><!--kg-card-begin: html--><script src="https://gist.github.com/Hameds/46c152286eca1c0b851f25cfb7060f36.js"></script><!--kg-card-end: html--><p>بیشتر کدی که نوشتم متدها متمرکز می‌شویم. متوجه شدید که mock ها را به ترتیبی که می‌خواهم متدها اجرا شوند arrange کرده‌ام.
