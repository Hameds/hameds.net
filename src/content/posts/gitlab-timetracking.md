<p>یکی از ویژگی‌های گیت‌لب، امکان <a href="https://docs.gitlab.com/ee/user/project/time_tracking.html" rel="noreferrer">ثبت زمان تخمین و اجرای کارها</a>ست. این کار به سادگی به کمک Quick Actionها انجام می‌شه. مثلاً اگر تخمین شما برای اجرای یک کار، یک ساعته به راحتی با کامنت گذاشتن عبارت زیر می‌تونید این زمان تخمینی رو به اون وظیفه (Issue) متصل کنید.</p><p><code>/estimate 1h</code></p><p>و برای اضافه کردن زمان واقعی اجرا هم از دستور سریع <code>spend</code> به شکل زیر می‌تونید استفاده کنید. در این مثال فرض می‌کنیم اجرای واقعی کار ۴۰ دقیقه طول کشیده.</p><p><code>/spend 40m</code></p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="/images/2023/10/time_tracking_sidebar_v13_12.png" class="kg-image" alt="" loading="lazy" width="268" height="239"><figcaption><span style="white-space: pre-wrap;">نمونه ثبت و نمایش رهگیری زمان در گیت‌لب</span></figcaption></figure><p>از اونجایی که از طریق commit message می‌شه بعضی کارها مثل مرتبط کردن و بستن اتوماتیک Issue در صورت Merge شدن commit رو در گیت‌لب انجام داد (در گیت‌هاب و سایرین هم کمابیش همینطوره)  <a href="https://gitlab.com/gitlab-org/gitlab-foss/-/issues/28044" rel="noreferrer">چندین ساله</a> که درخواست شده امکان استفاده از قابلیت‌های رهگیری زمان اجرای کارها هم به commit messageها اضافه بشه ولی فعلاً <a href="https://docs.gitlab.com/ee/user/project/quick_actions.html#commit-messages" rel="noreferrer">پشتیبانی محدودی</a> برای Quick Action وجود داره.</p><h3 id="%D8%B1%D8%A7%D9%87-%D8%AD%D9%84-%D8%AC%D8%A7%DB%8C%DA%AF%D8%B2%DB%8C%D9%86-quick-action">راه حل جایگزین Quick Action</h3><p>برای اینکه بتونید این قابلیت رو داشته باشید که همزمان با commit کردن اطلاعات دیگه‌ای هم اضافه کنید، یک راه حل استفاده از <a href="https://gitlab.com/gitlab-org/cli" rel="noreferrer">Gitlab CLI</a> به همراه تعریف یک function برای استفاده از قابلیت‌های CLI است.</p><p>بعد نصب CLI باید به Gitlab متصل بشید که بسته به اینکه از نسخه ابری استفاده می‌کنید یا نسخه self-hosted می‌تونید از روش OAuth یا Personal Access Token استفاده کنید که توضیحاتش در <a href="https://gitlab.com/gitlab-org/cli/-/blob/main/README.md" rel="noreferrer">Readme پروژه CLI</a> اومده.</p><p>حالا فرض کنید می‌خواهید همچین ساختاری داشته باشید:</p><p><code>gcptt 129 "actual commit message" 30m</code></p><p>که <code>gcptt</code> اسم اون function است که تعریف می‌کنیم و می‌تونه چند تا پارامتر ورودی داشته باشه:</p><p>پارامتر اول: شناسه Issue روی گیت‌لب.<br>پارامتر دوم: commit message که می‌خواهید ارسال کنید.<br>پارامتر سوم: زمانی که پیاده‌سازی کد طول کشیده و می‌خواهید در گیت‌لب ثبت کنید.</p><figure class="kg-card kg-code-card"><pre><code>gcptt() {n    git add --alln    git commit -m "resolve #$1 $2"n    git pushn    glab issue note $1 -m "/spend $3"n}</code></pre><figcaption><p dir="rtl"><span style="white-space: pre-wrap;">نمونه کد که می‌تونید توی فایل .zshrc قرار بدین.</span></p></figcaption></figure><p>حالا وقتی توی ترمینال روی repository هستین به راحتی می‌تونید با یک دستور هم commit و هم push و هم زمان رو ثبت کنید.</p><p>اگر بخواهید همزمان با push کردن یک Merge Request هم ایجاد کنید، به جای <code>git push</code> در خط چهارم می‌تونید از دستور <code>git push -o merge_request.create --set-upstream origin HEAD</code> استفاده کنید. من برای این هم یک alias تعریف کردم به اسم <code>gmr</code> و بعد به جای <code>git push</code> اون <code>gmr</code> رو قرار دادم.</p>
